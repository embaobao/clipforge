# 提案：GitHub 分发与检查升级能力

## 优先级

P0。ClipForge 当前仍处于桌面工具能力闭环阶段，稳定分发、版本检查、更新提示和回滚记录是继续迭代的前置能力。没有可靠分发链路时，后续剪贴板、多格式和搜索能力都很难给用户持续验证。

## 背景

ClipForge 需要从本地开发态走向可持续分发态。当前项目已有 `scripts/build-mac-release.sh`、Tauri v2 桌面壳和本地安装说明，但还缺少一条清晰的 GitHub Release 分发和更新检查链路：

- 发布产物、签名、校验文件、release notes 没有统一契约。
- 应用内还没有稳定的“检查更新”入口和更新状态模型。
- 自动升级、插件/能力 manifest 升级已在 `context-plugin-agent-runtime` 中被提及，但范围太大，需要先把应用分发和更新检查独立落地。
- 失败、回滚、禁用更新源、忽略版本等用户可控能力还没有定义。

## 目标

1. 以 GitHub Releases 作为第一阶段分发源，定义 macOS 安装包、校验文件、更新 manifest 和 release notes 的发布格式。
2. 增加应用内“检查更新”能力，支持手动检查、启动后延迟检查、更新可用提示、下载进度和失败状态。
3. 接入 Tauri updater 的签名更新路径，保证应用更新来自可信 artifact。
4. 建立本地更新状态记录：最近检查时间、当前版本、可用版本、忽略版本、失败原因、回滚记录。
5. 保持更新检查不阻塞快速面板启动，也不影响剪贴板监听。

## 非目标

- 不在第一阶段实现插件市场或远程插件自动安装。
- 不做静默强制更新。
- 不在第一阶段支持多渠道复杂灰度（只保留 stable / prerelease 的基础区分）。
- 不把 GitHub API token 写入客户端。
- 不解决 Windows/Linux 打包的全部细节，但设计必须保留跨平台字段。

## 用户价值

- 用户能从 GitHub Release 获取可信安装包。
- 用户能在应用内知道当前版本是否过期，并自己决定是否更新。
- 更新失败有可读原因，不会破坏剪贴板主路径。
- 后续内置能力 manifest、插件和 Agent adapter 更新可以复用同一套状态、日志和权限模型。

## 依赖关系

- 依赖：现有 Tauri v2 应用配置和构建脚本。
- 关联：`context-plugin-agent-runtime` 中的“自动升级能力规划”。
- 不阻塞：`search-filter-tags-filetypes`、`content-smart-format-decoder`。

## 成功标准

- GitHub Release 中能找到当前平台可用 artifact、签名和更新 manifest。
- 应用设置页或关于页能显示当前版本、最近检查时间和更新状态。
- 手动检查更新能区分：已是最新、发现新版本、网络失败、签名/manifest 不可信、当前平台无可用包。
- 更新检查日志写入本地结构化日志。
- 快速面板启动期间不等待网络请求。
