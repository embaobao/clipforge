# 提案：应用国际化与英文支持

## 优先级

P0.5。该能力应跟随 `github-release-update-distribution` 之后、公开分发前完成基础闭环。它不阻塞分发升级本身，但应早于格式支持、搜索增强和智能解析的大量 UI 文案落地，避免后续功能继续积累硬编码中文。

## 背景

ClipForge 当前默认使用中文界面，`index.html` / `settings.html` 声明为 `zh-CN`，React 组件、设置页、托盘菜单、状态提示、错误提示和 release 辅助页面中存在大量直接写死的中文或英文文案。

随着 GitHub Releases 分发、英文 README、跨平台用户和后续功能增加，应用需要具备基础国际化能力：

- 默认语言仍可保持中文，但必须能切换英文。
- 用户可见 UI 文案需要从硬编码迁移到稳定 key。
- 设置、快速面板、详情页、托盘菜单、更新提示、权限提示需要统一语言。
- 系统日志、诊断字段和开发文档不应被误纳入翻译范围。

## 目标

1. 建立轻量国际化基础设施，支持 `zh-CN` 和 `en-US`。
2. 设置页提供语言选择：跟随系统、中文、英文。
3. 用户可见 UI 文案通过稳定 key 获取，不再在新增功能里继续硬编码。
4. `html lang`、窗口标题、托盘菜单、更新提示、权限提示和空状态随语言切换。
5. 建立文案收口流程：开发期允许临时中文，提测/发版前必须补 key、补英文、检查残留。

## 非目标

- 不翻译用户剪贴板内容、标签、文件名、历史记录或用户输入。
- 不在第一阶段接入远程翻译平台。
- 不做复杂复数规则、日期货币本地化或 RTL 语言。
- 不把结构化日志字段、错误码、traceId、命令名、MCP tool 名翻译掉。
- 不要求一次性翻译所有历史文档。

## 用户价值

- 中文用户继续使用中文界面。
- 英文用户可以通过设置切换到可用英文界面。
- GitHub 分发后，新用户打开应用能获得一致的英文体验。
- 后续新增格式、搜索、智能解析功能时，有统一文案入口和检查流程。

## 依赖关系

- 前置关联：`github-release-update-distribution` 负责分发和更新入口，国际化需覆盖其用户可见文案。
- 后续约束：`file-image-clipboard-support`、`clipboard-multi-format-fidelity`、`search-filter-tags-filetypes`、`content-smart-format-decoder` 的新增 UI 文案必须接入本提案定义的 key。
- 可独立启动：不依赖多格式剪贴板、搜索筛选或智能解析实现。

## 成功标准

- 应用可在中文和英文之间切换，重启后保持选择。
- 快速面板、设置页、托盘菜单、更新入口和常见错误/空状态有中英文文案。
- `index.html` / `settings.html` 的 `lang` 能随当前语言或初始语言策略设置。
- 新增文案有 key 命名规则和检查脚本。
- 发版前能扫描出用户可见硬编码中文/英文残留，并允许白名单排除日志、测试、文档和用户内容。
