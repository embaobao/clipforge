# 提案：主面板与设置页视觉回归修复

## 优先级

P0。当前问题直接影响用户对主面板和设置页的可用性判断，必须先于继续扩展功能或继续推进长期布局规划处理。

## 背景

在推进设置页重构和顶部导航优化时，出现了几类不该发生的视觉/交互回归：

1. **主列表选中交互被改坏**
   - 原本列表有稳定的选中/激活交互。
   - 后续为了去掉“难看的虚线框”，错误扩大成重写 `.quick-row.active` / `.quick-row.selected` / `target-focus-ring`。
   - 结果改变了用户熟悉的列表选择反馈和行状态层级。

2. **顶部菜单样式被改坏**
   - `top-toolbar-menu` 只加了最小宽度和 gap，没有完整定义菜单背景、字号、行高、hover、header 对比。
   - 通用 dropdown 样式在浮动主面板里显得过大、过虚、层级不清。
   - 菜单 header 低对比，菜单项高度和灰底状态显得笨重。

3. **顶部工具栏优化越界**
   - 顶部工具栏需要解决拥挤和重叠，但不应顺手改变列表行状态、菜单视觉系统或其它交互。
   - 当前 CSS 中存在 P-FINAL 覆盖堆叠，容易继续产生局部修复引发其它区域回归。

4. **设置页组件化方向正确，但样式仍需收口**
   - 已引入 Animate UI Sidebar / Code Tabs 的本地组件。
   - 但设置页样式仍存在多段覆盖，需要确认最终样式只服务设置页，不影响主面板。

## 目标

1. 恢复主列表原有选中/激活交互，不再重写用户已接受的列表选择逻辑。
2. 修复顶部菜单，使其成为轻量、清晰、可点击的工具菜单。
3. 收窄顶部工具栏修复范围，只处理布局拥挤、搜索栏宽度、按钮尺寸和菜单入口。
4. 清理或标记危险 CSS 覆盖，避免继续用全局 P-FINAL 覆盖压问题。
5. 为设置页视觉收口建立边界：设置页样式只通过设置页容器和 Animate UI slot 生效。

## 非目标

- 不重新设计主列表。
- 不改变剪贴板行点击、键盘选中、复制、粘贴、多选逻辑。
- 不重写 `VirtualList`。
- 不引入新的菜单组件库。
- 不继续扩大设置页功能范围。
- 不处理 Context7 文档拉取和硬编码文案治理。

## 修复原则

1. **恢复优先**：被误改的主列表交互优先回到原状态。
2. **最小 CSS 面**：每次只改一个 surface，选择器必须绑定 `.app-shell .top-toolbar-menu` 或 `.settings-window-shell` 等明确边界。
3. **不碰热路径逻辑**：主列表选中、滚动、复制、粘贴只允许验证，不允许顺手重构。
4. **视觉修复要可截图验收**：菜单、列表、设置页必须能通过运行态截图或用户目视确认。
5. **任务完成后更新提案状态**：每修完一项，在本提案 tasks 里记录证据。

## 成功标准

1. 主列表行选中、hover、active、selected、多选状态恢复到本轮错误改动前的交互语义。
2. 顶部菜单：
   - 背景清晰，不是大面积模糊白块。
   - header 可读但不过重。
   - Trash / Settings 行高紧凑，hover 清晰。
   - 快捷键对齐，不挤压文字。
3. 顶部工具栏：
   - History / Favorites / Search / Agent / Menu 不重叠。
   - 搜索框保留可输入宽度。
   - 菜单按钮点击不触发拖拽。
4. 设置页：
   - Sidebar / Tabs / Code Tabs 样式只在设置页容器内生效。
   - 不污染主面板 dropdown、quick-row、toolbar。
5. `pnpm exec tsc --noEmit`、`node scripts/verify-settings-surface.mjs`、`pnpm openspec validate quick-panel-visual-regression-recovery --strict` 通过。

- 顶部菜单样式仍需按本提案补齐并验证。
